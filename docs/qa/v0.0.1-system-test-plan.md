# ManoMatika v0.0.1 — End-to-End System Test Plan

**Release under test:** ManoMatika **v0.0.1** = matika **v0.0.4** + eyerate **v0.0.4** + ahimsa **v0.0.1**
**Artifacts under test:** ahimsa-built installers (macOS DMGs + Windows EXE) attached to the matika v0.0.4 GitHub release.
**Closes:** manomatika/ahimsa#33

> This document is an **executable checklist** the tester (human) follows on **clean** target machines. Each functional case cites the QA issue(s) its result closes, so a passing case translates directly into checking off that issue. **Do not** run this against a dev checkout — the point is to validate the shipped installers on machines that have never run ManoMatika.

---

## 1. Scope, composition, and definition of "pass"

### What is under test
- The **matika v0.0.4** framework, packaged by ahimsa into standalone installers, **hosting the eyerate v0.0.4 applug** as the bundled reference AppLug.
- The **ahimsa v0.0.1** build pipeline's output: the DMG (macOS arm64 + x86_64) and EXE (Windows 11) installers, including their first-run initialization and unsigned-installer trust-prompt behavior.

### What is NOT under test here
- Source-tree / CI unit suites (already green per each repo's `validate`/test workflows).
- The ahimsa build pipeline's internal mechanics (validated when `build.yml` is dispatched against the v0.0.4 tags).
- Code signing / notarization — **explicitly out of scope for v0.0.1**; installers are unsigned by design (see §3).

### Definition of "pass" for v0.0.1
> **v0.0.4 scope note:** the Windows 11 target is **deferred** (no test hardware — see §2.1); v0.0.4 is gated on the **two macOS targets** only. The criteria below apply to every target that is **actually executed** (macOS arm64 + x86_64 for v0.0.4; Windows when manomatika/ahimsa#46 is run).

ManoMatika v0.0.1 **passes** when, on **each executed** clean-machine target:
1. The installer mounts/installs and the app launches past the unsigned-installer trust prompt.
2. First-run initialization completes (SECRET_KEY generated, DB migrated, eyerate plugin extracted) and the app serves its UI on `http://127.0.0.1:8000`.
3. The framework boots and **loads the eyerate applug**; the eyerate **Securities** page renders and its core CRUD + data-fetch + permission behavior works.
4. No **blocker** or **critical** defect is found (see §7 severity). Cosmetic/minor issues are filed but do not block the go decision.

A target is **not** required to be re-tested for an arch it cannot run (e.g. an Apple-silicon tester cannot run the x86_64 DMG natively); see the matrix for which physical machines are needed.

## 3. Unsigned-installer handling — REQUIRED, expected behavior (not a defect)

v0.0.1 installers are **not code-signed or notarized**. macOS **Gatekeeper** and Windows **SmartScreen** will warn. This is the documented v0.0.1 trust boundary (see the unsigned-installer limitation in the v0.0.4 release notes, manomatika/ahimsa#26). Treat the warning as **expected**; record `PASS` if the documented bypass lets the app run. Record `FAIL` only if the app cannot be launched at all by the steps below.

### macOS — Gatekeeper path (TC-A1 / TC-B1)
1. Double-click the app the first time → expect a block: *"… can't be opened because Apple cannot check it for malicious software."*
2. **Bypass:** Control-click (or right-click) the app icon → **Open** → in the dialog, click **Open** again. (Alternatively: System Settings → Privacy & Security → "Open Anyway".)
3. **Expected:** the app launches and is thereafter remembered as allowed.
4. **PASS** if the app runs after the control-click → Open path. **FAIL** if it remains blocked.

### Windows 11 — SmartScreen path (TC-C1)
1. Run the EXE → expect *"Windows protected your PC"* (Microsoft Defender SmartScreen).
2. **Bypass:** click **More info** → **Run anyway**.
3. **Expected:** the installer proceeds.
4. **PASS** if the installer runs after Run anyway. **FAIL** if it cannot be launched.

> If the tester's org policy hard-blocks unsigned binaries (no "Run anyway" / "Open Anyway" available), mark the affected suite **BLOCKED** and note the policy — this is an environment limitation, not a product defect, but it must be recorded for the go/no-go.

## 4. Install / launch / first-run cases

Common preconditions for **every** target:
- A **clean** machine (or fresh VM/user account) that has never run ManoMatika. No `~/matika/` directory (macOS/Linux) or `%USERPROFILE%\matika\` (Windows) present.
- Port **8000** on `127.0.0.1` is free.
- The correct artifact for the target downloaded from the matika v0.0.4 GitHub release:
  - macOS arm64 → `*-macos-arm64.dmg`
  - macOS x86_64 → `*-macos-x86_64.dmg`
  - Windows 11 → `*-windows-x86_64.exe`

### TC-A2 / TC-B2 — DMG mount + install to /Applications (macOS) — closes manomatika/matika#32 (arm64) / manomatika/matika#33 (x86_64); contributes to manomatika/ahimsa#15 / #16
**Preconditions:** common, above. Gatekeeper bypass (TC-A1/B1) understood.
**Steps:**
1. Double-click the downloaded `.dmg` → the disk image mounts and a Finder window opens.
2. Confirm the window shows the **app icon** and an **Applications** shortcut (drag-to-install layout).
3. Drag the app onto the **Applications** shortcut.
4. Eject the DMG.
5. Confirm the app now appears in `/Applications`.
**Expected:** DMG mounts cleanly; drag-install copies the full app bundle to `/Applications`; no copy errors.
**Result:** ☐ PASS ☐ FAIL ☐ BLOCKED **Notes:** _______________

### TC-C2 — EXE install (Windows 11) — closes manomatika/matika#34; contributes to manomatika/ahimsa#19
**Preconditions:** common, above. SmartScreen bypass (TC-C1) understood.
**Steps:**
1. Run the `*-windows-x86_64.exe` installer; pass SmartScreen (TC-C1).
2. Step through the Inno Setup wizard (accept defaults unless testing a custom path).
3. Confirm the install completes without error and creates the application directory + Start-menu entry.
**Expected:** the installer lays down the **full one-dir bundle** (all framework + eyerate plugin assets present, not a single bare exe); install completes; launch entry created.
**Result:** ☐ PASS ☐ FAIL ☐ BLOCKED **Notes:** _______________

### TC-A3 / TC-B3 / TC-C3 — First launch + first-run initialization — closes the install issues above (manomatika/matika#32/#33/#34, manomatika/ahimsa#15/#16/#19)
First-run init is performed by the matika launcher (per manomatika/matika#28 launcher behavior).
**Preconditions:** app installed (TC-*2). No prior `~/matika/` / `%USERPROFILE%\matika\`.
**Steps:**
1. Launch the app (macOS: control-click → Open the first time; Windows: Start-menu entry).
2. Wait for first-run init to complete. Verify the data directory is created:
   - macOS: `~/matika/` exists.
   - Windows: `%USERPROFILE%\matika\` exists.
3. Verify first-run artifacts inside that directory:
   - `.env` exists and contains a generated **SECRET_KEY** (non-empty, not a placeholder).
   - The application **database** file exists and has been migrated (tables present — the app reaching a working login/UI is sufficient evidence; do not hand-inspect schema).
   - A `plugins/` directory exists with **`eyerate/`** extracted into it (the bundled applug).
   - The `.initialized` sentinel file exists (init ran to completion).
4. Confirm the app **serves its UI**: a browser opens (or open one manually) to `http://127.0.0.1:8000` and the ManoMatika login/landing page renders.
5. **Port-conflict sub-check (optional but recommended):** with the app running, launch a second instance → expect a clear user-facing "port 8000 in use" error rather than a silent crash. Record under Notes.
**Expected:** init completes once, populates `~/matika/` as above, and the UI is served on `127.0.0.1:8000`. Re-launching later does **not** re-run init (sentinel respected).
**Result:** ☐ PASS ☐ FAIL ☐ BLOCKED **Notes:** _______________

## 5. Functional cases — framework + eyerate applug

### TC-A5 / TC-B5 / TC-C5 — eyerate Securities page functional — closes manomatika/eyerate#17 (macOS) / manomatika/eyerate#18 (Windows)
This is the eyerate functional acceptance check (manomatika/eyerate#17 on macOS, manomatika/eyerate#18 on Windows).
**Preconditions:** logged in as an **Admin** user (TC-*4). Have a second **non-admin (User)** account available, or create one, for the permission sub-check.
**Steps:**
1. Navigate to the **EyeRate** hub → open the **Securities** maintenance page.
2. **Render:** confirm the Securities page loads and its table/controls render without error.
3. **Create:** add a security (e.g. ticker `AAPL`) → confirm it is created and appears in the list.
4. **Read/Update:** open the created security, edit a field, save → confirm the change persists after reload.
5. **Data fetch:** trigger the market-data fetch (Yahoo Finance) for the security → confirm data is retrieved and displayed. *(Requires outbound internet; if the test machine is offline, mark this sub-step BLOCKED and note it.)*
6. **Delete:** delete the security → confirm it is removed from the list.
7. **Permissions:** log out, log in as the **non-admin User**, attempt to reach the Securities **admin** route/page → confirm access is **denied** (User cannot reach admin-only Securities maintenance). Then confirm the User sees only what their role permits.
**Expected:** Securities page renders; full CRUD works; market-data fetch succeeds (online); RBAC denies the admin page to non-admin users.
**Result:** ☐ PASS ☐ FAIL ☐ BLOCKED **Notes:** _______________

## 7. What to do on failure

- **File each defect as a separate issue in the repo that owns the fault**, per manomatika/ahimsa#33's standing instruction ("Add discovered defects as separate issues in the relevant repo"):
  - Framework / launcher / install-init / packaging-of-matika fault → **manomatika/matika**.
  - eyerate Securities / applug-specific fault → **manomatika/eyerate**.
  - Installer build / DMG / Inno Setup / release-pipeline fault → **manomatika/ahimsa**.
- **Reference this plan** in the new issue: cite the failing case ID (e.g. `TC-C5`), the target, and link to `docs/qa/v0.0.1-system-test-plan.md` plus manomatika/ahimsa#33.
- **Severity guidance:**
  - **Blocker** — app cannot install or launch on a target; UI does not serve; eyerate does not load. → NO-GO.
  - **Critical** — core Securities CRUD or RBAC broken; data loss; init corrupts `~/matika/`. → NO-GO.
  - **Major** — a functional path fails but a workaround exists. → triage before GO.
  - **Minor / cosmetic** — labels, layout, copy. → file, but does not block GO.
- Do **not** edit/close the QA issues until their mapped cases pass; do **not** close manomatika/ahimsa#33 until the full matrix is executed and the go/no-go is recorded.
